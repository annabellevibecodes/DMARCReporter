SPF — Sender Policy Framework

SPF lets a domain owner publish a list of IP addresses that are authorised to send mail on behalf of that domain. When a receiving mail server gets a message, it looks up the SPF record in DNS for the domain found in the envelope from (also called the return-path or bounce address) and checks whether the sending IP is on the list.

How it works:

  1. Sender publishes a TXT record at the domain root, e.g.
    v=spf1 include:_spf.google.com ip4:203.0.113.10 ~all
  2. Receiving server extracts the envelope-from domain and performs a DNS lookup.
  3. If the sending IP matches, SPF passes; otherwise it fails.

Common SPF qualifiers:

QualifierMeaning
+all (or just all)Pass — allow all (not recommended)
~allSoftFail — treat unlisted senders with suspicion
-allFail — reject unlisted senders
?allNeutral — no policy statement

Limitation: SPF breaks when mail is forwarded, because the forwarding server's IP is no longer in the original domain's SPF record. This is one reason DKIM and DMARC exist.

DKIM — DomainKeys Identified Mail

DKIM adds a cryptographic signature to outgoing messages. The signature is tied to a domain and travels with the email, so it survives forwarding. The receiving server retrieves the matching public key from DNS and verifies the signature.

How it works:

  1. The sending mail server signs the message with a private key and adds a DKIM-Signature header, referencing a selector and domain, e.g.
    DKIM-Signature: v=1; a=rsa-sha256; d=example.com; s=mail; ...
  2. The receiving server looks up the public key at <selector>._domainkey.<domain> in DNS.
  3. If the signature verifies, DKIM passes.

What DKIM protects: It proves the message body and selected headers were not tampered with in transit, and that the signer controls the signing domain.

What DKIM does not protect: The visible From header — that is handled by DMARC alignment (see below).

DMARC — Domain-based Message Authentication, Reporting & Conformance

DMARC builds on SPF and DKIM by tying both checks to the domain visible in the email's From header — the address your users actually see. It tells receiving servers what to do when authentication fails, and instructs them to send aggregate reports back to you.

A DMARC record looks like:

_dmarc.example.com  TXT  "v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com; pct=100;"

Key tags:

TagMeaning
vProtocol version — always DMARC1
pPolicy for the domain: none, quarantine, or reject
spPolicy for subdomains (falls back to p if absent)
ruaAddress(es) to receive aggregate reports (this dashboard reads these)
rufAddress(es) to receive forensic (failure) reports
pctPercentage of failing mail to apply the policy to (default: 100)
adkimDKIM alignment mode: r (relaxed, default) or s (strict)
aspfSPF alignment mode: r (relaxed, default) or s (strict)
foFailure reporting options (controls when ruf reports are sent)

A message passes DMARC when at least one of the following is true:

  • SPF passes and the envelope-from domain aligns with the header-from domain.
  • DKIM passes and the signing domain (d=) aligns with the header-from domain.
DMARC Policies

The p= tag controls what happens to mail that fails DMARC:

PolicyEffect on failing mailTypical use
none Deliver normally — no action taken on failures Monitoring phase; safe to start here while you identify all legitimate senders
quarantine Move to spam/junk folder Transition phase once you are confident your legitimate mail passes
reject Reject at the SMTP level — mail is not delivered Full enforcement; best protection against spoofing

The recommended path is none → quarantine → reject. Move to the next level only when your DMARC reports show consistently high pass rates from all legitimate sources.

Alignment

Alignment is the check that connects SPF/DKIM results to the domain in the visible From header. Without alignment, a spammer could set up SPF/DKIM for their own domain while spoofing yours in the From header.

ModeSPF alignment ruleDKIM alignment rule
Relaxed (r, default) Envelope-from domain must be the same registered domain as header-from (subdomains are allowed) DKIM d= domain must share the same registered domain as header-from
Strict (s) Envelope-from domain must match header-from domain exactly DKIM d= domain must match header-from domain exactly

Example: with relaxed alignment, a message From user@example.com with DKIM signed by mail.example.com still passes, because both share the registered domain example.com. Under strict mode it would fail.

Disposition

The disposition reported in an aggregate report indicates what the receiving mail server actually did with a message, independent of what the policy requests:

ValueMeaning
noneNo action was taken (delivered normally)
quarantineMessage was placed in spam/junk
rejectMessage was rejected at the SMTP level

When the domain's policy is none, receivers will almost always report none regardless of whether authentication passed or failed. Use the pass/fail columns to gauge your actual authentication health during the monitoring phase.

BIMI — Brand Indicators for Message Identification

BIMI is an emerging standard that allows domain owners to display a verified brand logo next to their emails in supporting mail clients (Gmail, Apple Mail, Yahoo Mail, etc.). It requires a valid DMARC policy of quarantine or reject.

DNS record:

default._bimi.example.com  TXT  "v=BIMI1; l=https://example.com/logo.svg; a=https://example.com/cert.pem;"

Key tags:

TagMeaning
vVersion — always BIMI1
lURL of the SVG logo (must be a square SVG Tiny PS file)
aURL of a Verified Mark Certificate (VMC) — required by most major providers to display the logo

Without a VMC (issued by an accredited Certificate Authority), most mail clients will not display the BIMI logo even if the record is present.

MTA-STS — Mail Transfer Agent Strict Transport Security

MTA-STS is a mechanism that allows domains to declare that inbound SMTP connections must use TLS and that the TLS certificate must be valid. It prevents downgrade attacks where an attacker strips TLS from a mail connection.

How it works:

  1. Publish a DNS record: _mta-sts.example.com TXT "v=STSv1; id=20240101000000Z;"
  2. Host a policy file at https://mta-sts.example.com/.well-known/mta-sts.txt listing your MX hosts and the enforcement mode.
  3. Sending servers that support MTA-STS will fetch and cache your policy and enforce TLS when delivering mail to your domain.

Policy modes:

ModeBehaviour
enforceSending server must deliver over valid TLS or fail the delivery
testingPolicy is observed but failures are only reported, not enforced
nonePolicy is being withdrawn

The id tag in the DNS record acts as a cache-busting mechanism — change it whenever you update the policy file to prompt sending servers to refresh their cached copy.

Reading This Dashboard

Pass vs Fail

A message row is counted as passed when it passes DMARC (i.e. SPF or DKIM passed with correct alignment). It is counted as failed when neither check passed with alignment. A high failure rate from a source you recognise usually means misconfigured SPF/DKIM; from an unknown source it may indicate spoofing.

Pass rate

Shown as a percentage of messages from that source that passed DMARC. A legitimate sending service should be at or near 100%. Values well below 100% warrant investigation.

Envelope From vs Header From

The envelope from (return-path) is used for SPF and for bounce delivery. The header from is the address displayed to recipients. DMARC requires alignment between these two. A mismatch highlighted in the report means the sending service is using a different return-path domain, which is common with third-party senders (ESPs, CRMs). Those senders need to either publish DKIM signed with your domain or be added to your SPF record.

Source IPs

Each row in an aggregate report represents one source IP sending mail that claims to be from your domain. Unexpected IPs you do not recognise could indicate a misconfigured service, a compromised account, or an active spoofing attempt.

Data Enrichment

Use the Enrich page to resolve reverse DNS, WHOIS organisation, and country data for all source IPs, and to check DMARC, BIMI, and MTA-STS records for all known domains. Enriched data is cached and used throughout the dashboard.
